# 提示词工程师 (Prompt Engineer)

> 将模糊的人类需求转化为机器可完美执行的结构化指令——一位不急于执行、先深度理解需求的 AI 提示词架构师。

## 概述

这是一个 Prompt 模板（角色定义），将 AI 转变为一个高级 Prompt 架构师。面对用户模糊的需求时，它不会直接写提示词，而是执行严格的三阶段流程：需求对齐与批判 → 任务分解 → 提示词工程。核心价值是**在做事之前先把事情想清楚**。

## 核心原理

### 三阶段设计哲学

大多人写提示词的问题是：拿到需求就直接写，写完发现不好用，反复改。Prompt Engineer 的解决方案是**把"写提示词"本身变成一个工程化流程**：

```
模糊需求 → [对齐与批判] → 精确定义 → [任务分解] → 原子任务树 → [工程撰写] → 高质量 Prompt
              ↑ 反复追问                          ↑ 动态调整                    ↑ 结构化输出
```

每个阶段之间有明确的门控——必须用户确认后才能进入下一阶段，确保不会在方向不明时就盲目动笔。

## 工作流（三阶段）

### 第一阶段：需求对齐与批判 (Alignment)

| 动作 | 说明 |
|------|------|
| 重述理解 | 用自己的话精准概括用户目标 |
| 批判与推演 | 指出逻辑漏洞、缺失信息或风险 |
| 完善信息体系 | 至少 4 个追问，模糊部分不断追问直到彻底确定 |
| 发散建议 | 基于专业知识提出优化建议 |
| **门控** | 必须询问"我们是否达成了共识？" |

### 第二阶段：任务分解 (Decomposition)

| 动作 | 说明 |
|------|------|
| 拆解 | 将宏大目标拆为线性、可执行的原子任务树 |
| 输出 | Task 1.1, Task 1.2... 清晰任务列表 |
| **门控** | 询问对拆解逻辑的意见，确认后进入撰写 |

### 第三阶段：提示词工程 (Engineering)

- 针对确认的任务列表撰写 System Prompt
- 使用 Markdown 结构（Role, Context, Constraints, Steps）
- 输出在独立代码块中，一键复制
- 运用 CoT（思维链）、Few-Shot（少样本）等高级技巧

## 适用场景

- 需要写一个复杂的 System Prompt 但不知道从哪下手
- 有一个模糊的想法（"我想要一个帮我写周报的 AI"），需要别人帮你理清
- 现有 Prompt 效果不好，需要专业诊断和重写
- 需要批量生产高质量、结构一致的 Prompt 模板
- 学习 Prompt Engineering 方法论——看它怎么拆解你的需求本身就是学习

## 优点

- **防急于执行**：核心思维原则第一条就是"深度理解而非表面顺从"
- **门控机制**：每阶段必须用户确认，避免方向跑偏
- **批判性思维**：主动发现用户逻辑盲点，不盲从
- **工程化输出**：代码块一键复制，Markdown 结构化，直接可用
- **可学习性**：三阶段流程本身就是一套 Prompt Engineering 方法论

## 可能的问题

- 三阶段流程对简单需求（"帮我写一个翻译 prompt"）过度展开，显得啰嗦
- "至少 4 个追问"有时显得刻意——简单需求不需要这么多问题
- 对于已经很清楚自己需求的高级用户，门控机制可能是一种打扰
- 最终 Prompt 质量受限于运行此模板的底层模型能力

## 与 skill 的区别

这不是一个 Claude Code Skill（没有 YAML frontmatter 和触发机制），而是一个可复用的 **Prompt 模板**。使用方式：
- 直接复制角色定义到对话开头
- 或改造为 skill（添加 frontmatter 和触发词）
- 适合作为"元提示词"——用来生成其他 Prompt 的 Prompt
